Framing: this article isn't about prompt-crafting tricks — it's about a shift already visible in how the best engineers use coding agents: instead of typing prompts by hand, they write the *loop* that keeps prompting the agent for them.

Point people at the repo now if you want them following along live: github.com/d48/loop-engineering-demo — stdlib-only Python, no API keys needed for any of the four demos.

Definition

Loop engineering: designing the system that prompts the agent

"I don't prompt Claude anymore. I have loops running that prompt Claude and figuring out what to do. My job is to write loops."
— Boris Cherny, Anthropic (creator of Claude Code)
  • Also promoted by Peter Steinberger (OpenClaw) and Addy Osmani (formerly Google).
  • The core move: replacing yourself as the person who prompts the agent, with a system that does it on your behalf.
manual prompting: you -> prompt -> agent -> result (repeat by hand, forever) loop engineering: you -> loop -> agent -> result ^ runs itself

Cherny's quote is the cleanest one-line definition in the piece. The shift he's describing: engineering effort moves from crafting individual prompts to designing the *system* that decides what to prompt next, when to stop, and how to recover.

This is the article's thesis in miniature — everything that follows (Ralph, /goal, triggers, cron) is a specific shape that "the loop" takes.

Where it began

The "Ralph Wiggum" loop — Geoffrey Huntley, mid-2025

while :; do cat PROMPT.md | claude-code ; done
  • Named after the naive Simpsons character who keeps offering to help — an agent that just keeps trying, restarted fresh each time.
  • Rationale: mid-2025 context windows topped out around ~200,000 tokens — breaking a big task into many small fresh runs worked around that ceiling.
"Engineers are still needed. There is no way this is possible without senior expertise guiding Ralph."
— Geoffrey Huntley
python3 demo/run.py 1 — a plan file survives across a dozen fresh-context ticks

Huntley's original post ("Ralph Wiggum as a software engineer") is the article's cited origin point for the whole loop-engineering trend, about a year before this piece was written.

His own caveat is worth reading aloud: this is not "fire and forget" automation that replaces engineers — it still requires someone senior enough to write the plan and recognize when Ralph is going sideways.

DEMO 1 runs the exact mechanic: PLAN.md is the only thing that survives between iterations; the "model" is invoked completely fresh each tick, exactly like a new `claude-code` process reading PROMPT.md from scratch.

How the Ralph loop actually works

Fresh context every tick; the plan file is the only memory

The circular motif here is deliberate: unlike a linear pipeline, the Ralph loop has no "next step" in the conventional sense — it's the same five moves, repeated, with the plan file as the one piece of state that survives the restart.

Subagents get spawned only when a plan item is big enough to warrant delegation — the article notes this keeps runs from blowing the context budget on any single tick.

The loop matures

"Ship working code while you sleep" — Matt Pocock's dynamic Kanban

  • Late 2025: better models made the Ralph loop reliable enough to run unattended overnight.
  • Pocock's version keeps a master PRD as a live backlog — a "dynamic Kanban" the loop works through on its own.
agent prompt, each tick: 1. pick next feature (priority) 2. pnpm test # must pass 3. update master tracker 4. log to progress.txt 5. git commit

Same five-step shape as the Ralph loop's own cycle — pick, work, verify, record, restart — just with a real test runner and version control standing in as the goal check.

This is the article's bridge between "Ralph as a hacker's trick" and "Ralph as production practice." Pocock's contribution was operational discipline: a real backlog, a real test gate (pnpm test), and a real commit per completed item — nothing conceptually new over Huntley's loop, but proof it scales to real overnight work.

The loop becomes a first-class feature

/goal ships across every major harness within six months

  • By May, running a Ralph-style loop needed one command in a mainstream harness — no more hand-rolled while-loops.
  • Claude Code's /goal: a small, fast model checks the completion condition after every turn and auto-clears the goal once it's met. An earlier March feature, /loop, already handled scheduling agents at intervals.

Six months from "one engineer's shell one-liner" to "a first-class command in every mainstream coding agent" — that speed is itself part of the article's point about how fast this category moved.

Claude Code's architecture is worth spelling out: the completion check runs on a *separate*, cheaper/faster model than the one doing the work — cheap verification on every turn, not just at the end.

Goal vs. prompt

"Do this next thing" vs. "keep working until this is true"

PROMPT "make checkout faster" -> one edit, ships, done -> nobody re-checks it GOAL "reduce p95 checkout latency below 120ms on the checkout benchmark while keeping the correctness suite green" -> loops with a real check -> stops only when both hold

This exact wording is Codex's own published example for its Goals feature — not a hypothetical.

python3 demo/run.py 2 — same benchmark, prompt mode ships a broken "fix"; goal mode doesn't stop until it's real

This is the single clearest technical definition in the whole article. A prompt is really just a goal with a budget of one tick and no verification step — that reframing is the key insight for engineers in the audience.

DEMO 2 makes the risk concrete: prompt mode takes the obvious shortcut (skip a lookup, "simplify" the tax math while it's in there) and ships it unverified — it hits the latency target but silently breaks correctness. Goal mode iterates, checking real p95 latency AND a real correctness suite after every tick, and only stops once both are genuinely true.

What a goal needs to actually work

Codex Goals: files, logs, tests, lifecycle, budget

The article frames this as the "harness" work behind a seemingly simple feature: /goal isn't just a completion-check prompt bolted onto the model — it needs real durable state, a real test runner it can call, controls a human can use to intervene, and a hard budget so a stuck loop doesn't run (and spend) forever.

The two shapes almost everyone reaches for

Triggers and cron jobs (~210 developer survey responses)

triggers

  • Agent wakes on an event: an error logged, a ticket filed, a piece of customer feedback.
  • Pre-AI parallel: a webhook firing a Slack bot, or a Zapier/n8n automation.

cron jobs

  • Agent wakes on a schedule — nightly, hourly, every 30 minutes.
  • Functionally identical to a traditional cron job; only the payload changed.
"Strategic workflows that are repeatable and automatable can be done so with an agent."
— Oded Messer, director of engineering
python3 demo/run.py 3 — real error events, deduped into PRs, with a stuck-PR notification

The article is candid about the critique here too: some engineers see "loop engineering" as a rebrand of automation that predates LLMs by a decade. What's actually new is the payload each trigger fires — a whole reasoning agent instead of a fixed script — not the trigger mechanism itself.

DEMO 3 mirrors Ivan Pantić's PR-from-Sentry-errors workflow directly: real events land in an inbox, get deduped by signature so three occurrences of one bug become one PR (not three), and a stuck PR triggers a real notification after enough ticks with no review.

What the article says about the downsides (partially paywalled)

Disappointment, "tokenmaxxing," and a live open question

  • Some developers try looping, hit agent drift, and go back to a human in the loop — plainly better results for them.
  • "Tokenmaxxing": token-hungry loops running up real API cost with no guardrail.
  • Max Kanat-Alexander's hypothesis: loops may be a temporary infrastructure hack, not a durable abstraction.
  • Open question the article poses directly: does context engineering end up mattering more than loop specialization?

Takeaways for working engineers

What to steal for your own loops

  • A prompt is a loop with a budget of one and no verification. Naming that distinction changes how you design agent-driven automation.
  • Persistent state beats conversation memory. A plan file that survives a fresh restart is more robust than a long-lived context window.
  • Never trust a claimed outcome — compute it. Every real loop in this deck ends with an independent, re-run check, not a stated one.
  • Budgets are a feature, not a limitation. A goal that can fail honestly within N ticks is safer than one that runs forever.
  • Triggers and cron aren't new — the payload is. The mechanism is decades old; what changed is what runs when it fires.
